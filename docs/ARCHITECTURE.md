# ARCHITECTURE — 360tur-landing

Документ фиксирует архитектурные решения и их обоснования.
Обновляется при значимых изменениях в структуре проекта.

---

## Тип приложения

SPA (Single Page Application), статический билд, без бэкенда.
Хостинг — статический. Целевая платформа (см. `TODO.md`, Приоритет 9): **VPS, деплой в подпапку** (`/virtualtour360/landing/`); поведение параметризовано через `VITE_BASE_PATH` (см. ниже «Деплой и базовый путь»).

## Сборщик

**Vite** (не Create React App).

Причина: шаблон уже использует Vite. CRA устарел и не поддерживается.

**Конфиг:** `vite.config.js` в корне репозитория.

## CSS

**Tailwind CSS** через PostCSS.
Кастомные токены (цвета, шрифты) — только в конфиге Tailwind, не в компонентах напрямую.

**Конфиг:** `tailwind.config.js` в корне репозитория. Палитра и шрифтовые семьи зафиксированы из `docs/DESIGN_BRIEF.md`.

**Глобальные стили и кастомные утилиты** — в `src/index.css` под директивами `@layer base`, `@layer components`, `@layer utilities`. Пример: `.text-glow-surface` (белый text-shadow для Hero-заголовка) — в `@layer utilities`. Inline-utility (`[text-shadow:...]`) допустимо для одноразовых случаев; повторно используемые эффекты — в `index.css`.

## Маршрутизация

**React Router**, `BrowserRouter` (чистые URL без `#`).
Лендинг одностраничный, но роутер оставлен на случай появления отдельных страниц (условия, политика и т.д.).

`BrowserRouter` инициализируется с `basename={import.meta.env.BASE_URL}` — это позволяет корректно работать роутеру при деплое в подпапку (см. «Деплой и базовый путь»).

Важно: `BrowserRouter` требует, чтобы хостинг умел возвращать `index.html` на произвольные пути (Netlify / Vercel — по умолчанию; GitHub Pages / Apache / nginx — потребуют SPA-fallback). Это учитывается при выборе платформы деплоя.

## Структура `src/`

```
src/
├── assets/       # импортируемые статические ресурсы (картинки, иконки), попадают в JS-бандл (хешируются)
├── components/   # переиспользуемые UI-компоненты (flat-структура)
├── contexts/     # React-контексты для глобального состояния (например, LanguageContext)
├── locales/      # словари переводов по языкам (ru.js, uz.js, en.js)
├── pages/        # экраны, соответствующие маршрутам React Router
├── App.jsx       # корневой компонент, роутер
├── main.jsx      # точка входа, монтирование в DOM, провайдеры (BrowserRouter, LanguageProvider)
└── index.css     # директивы Tailwind + @layer base/components/utilities
```

### Компоненты (`src/components/`)

Flat-структура — без вложенных папок, пока компонентов мало.
При росте числа компонентов — группировать по фиче: `components/hero/`, `components/contact/`.

### Страницы (`src/pages/`)

Отдельная папка от `components/` даже при одностраничности: `pages/` соответствуют маршрутам роутера (вспомогательные страницы вроде политики конфиденциальности, условий), `components/` — переиспользуемые блоки внутри страниц.

### Контексты (`src/contexts/`)

React Context API для глобального состояния, к которому обращается много компонентов из разных мест (в первую очередь — текущий язык интерфейса).

### Локали (`src/locales/`)

См. раздел «Интернационализация (i18n)».

## Ассеты

**Правило:**
- `src/assets/` — небольшие изображения/иконки, импортируемые из JSX (`import logo from '../assets/logo.svg'`). Vite хеширует их и кладёт в билд с контролем кеширования. Подходит для иконок, лого, мелких иллюстраций.
- `public/` — статические файлы, доступные по фиксированному пути от корня сайта (`/...`), без обработки сборщиком. Подходит для тяжёлой статики (фото высокого разрешения), `favicon.ico`, `robots.txt`, `sitemap.xml`, OG-изображений, **встраиваемых через iframe Marzipano-туров**.

**Текущая раскладка:**
- `public/imgs/` — фотографии (например, фон Hero `bedroom_cam_001_res_4096x2048_00.jpg`).
- `public/tours/<name>/` — папки с экспортированными Marzipano-турами (по одному туру на подпапку, со своими `vendor/`, `tiles/`, `data.js`, `index.html`).

**Префикс базового пути для ассетов из `public/`:** при ссылке на файл из `public/` в JSX/CSS-in-JS пишется относительный путь без ведущего слеша (`'imgs/...'`, `'tours/.../index.html'`), и подставляется через `${import.meta.env.BASE_URL}`. Это нужно для корректной работы при деплое в подпуть (см. «Деплой и базовый путь»). Пример:

```jsx
<div style={{ backgroundImage: `url("${import.meta.env.BASE_URL}imgs/hero.jpg")` }} />
<iframe src={`${import.meta.env.BASE_URL}tours/demo/index.html`} />
```

## Деплой и базовый путь

Базовый путь приложения (`base` в Vite) параметризован через переменную окружения `VITE_BASE_PATH`. Это позволяет одной кодовой базой собирать билды как для корня домена, так и для деплоя в подпапку.

- `vite.config.js`: `base: process.env.VITE_BASE_PATH ?? '/'`.
- `BrowserRouter` использует `basename={import.meta.env.BASE_URL}` (значение `BASE_URL` берётся Vite из `base`).
- Все ссылки на ассеты из `public/` префиксуются `${import.meta.env.BASE_URL}` (см. «Ассеты»).

**npm-скрипты:**

| Команда | Назначение | Базовый путь |
|---------|------------|--------------|
| `npm run dev` | dev-сервер | `/` |
| `npm run build` | дефолтная production-сборка | `/` |
| `npm run build:prod` | alias для `build` | `/` |
| `npm run build:vps` | сборка под VPS-подпуть | `/virtualtour360/landing/` (через `cross-env VITE_BASE_PATH=...`) |
| `npm run preview` | локальный просмотр dist/ | значение из последнего `build*` |

`cross-env` нужен, чтобы установка `VITE_BASE_PATH=...` работала одинаково в Windows-PowerShell и в bash.

**Требование к серверу:** SPA-fallback (раздача `index.html` на любые пути под базовым путём). Для nginx — типовое правило `try_files $uri /<base-path>/index.html;`.

## Интернационализация (i18n)

Языки: **ru / uz / en** (русский — основной).

Подход: **собственная реализация без внешних библиотек** (без `i18next`, `react-intl` и т.п.).

- Словари — плоские JS-объекты в `src/locales/{ru,uz,en}.js`.
- Текущий язык и функция перевода — через React Context (`LanguageContext`) + хук `useLanguage()`.
- Выбор языка — компонент `LanguageSwitcher` (пишет в Context).

Обоснование: лендинг содержит десятки, а не тысячи строк; сложные возможности `i18next` (pluralization, ICU-формат, lazy loading namespaces) избыточны. Если в будущем потребуется плюрализация/сложная интерполяция — возможна миграция на `i18next` (замена Context на провайдер `i18next` + ключи в JSON).

## Управление состоянием

- **Локальное состояние** компонентов — `useState` / `useReducer`.
- **Глобальное состояние** — React Context API.
  - Сейчас только `LanguageContext`.
- **Внешние библиотеки (Redux, Zustand, Jotai, Recoil) не используются.**

Обоснование: объём глобального состояния мал (язык, возможно — открытое мобильное меню). Встроенных средств React достаточно.
