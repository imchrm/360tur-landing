# ARCHITECTURE — 360tur-landing

Документ фиксирует архитектурные решения и их обоснования.
Обновляется при значимых изменениях в структуре проекта.

---

## Тип приложения

SPA (Single Page Application), статический билд, без бэкенда.
Хостинг — статический (конкретная платформа определяется в `TODO.md`, Приоритет 6).

## Сборщик

**Vite** (не Create React App).

Причина: шаблон уже использует Vite. CRA устарел и не поддерживается.

**Конфиг:** `vite.config.js` в корне репозитория.

## CSS

**Tailwind CSS** через PostCSS.
Кастомные токены (цвета, шрифты) — только в конфиге Tailwind, не в компонентах напрямую.

**Конфиг:** `tailwind.config.js` в корне репозитория. Палитра и шрифтовые семьи зафиксированы из `docs/DESIGN_BRIEF.md`.

## Маршрутизация

**React Router**, `BrowserRouter` (чистые URL без `#`).
Лендинг одностраничный, но роутер оставлен на случай появления отдельных страниц (условия, политика и т.д.).

Важно: `BrowserRouter` требует, чтобы хостинг умел возвращать `index.html` на произвольные пути (Netlify / Vercel — по умолчанию; GitHub Pages — потребует SPA-fallback через 404.html). Это учитывается при выборе платформы деплоя.

## Структура `src/`

```
src/
├── assets/       # импортируемые статические ресурсы (картинки, иконки), попадают в билд
├── components/   # переиспользуемые UI-компоненты (flat-структура)
├── contexts/     # React-контексты для глобального состояния (например, LanguageContext)
├── locales/      # словари переводов по языкам (ru.js, uz.js, en.js)
├── pages/        # экраны, соответствующие маршрутам React Router
├── App.jsx       # корневой компонент, роутер, провайдеры
├── index.jsx     # точка входа, монтирование в DOM
└── index.css     # глобальные стили + директивы Tailwind
```

### Компоненты (`src/components/`)

Flat-структура — без вложенных папок, пока компонентов мало.
При росте числа компонентов — группировать по фиче: `components/hero/`, `components/contact/`.

### Страницы (`src/pages/`)

Отдельная папка от `components/` даже при одностраничности: `pages/` соответствуют маршрутам роутера (вспомогательные страницы вроде политики конфиденциальности, условий), `components/` — переиспользуемые блоки внутри страниц.

### Контексты (`src/contexts/`)

React Context API для глобального состояния, к которому обращается много компонентов из разных мест (в первую очередь — текущий язык интерфейса).

### Локали (`src/locales/`)

См. раздел «Интернационализация (i18n)».

## Ассеты

**Правило:**
- `src/assets/` — изображения/иконки, импортируемые из JSX (`import logo from '../assets/logo.svg'`). Vite хеширует их и кладёт в билд с контролем кеширования.
- `public/` — статические файлы, которые должны быть доступны по фиксированному пути без обработки сборщиком (`favicon.ico`, `robots.txt`, `sitemap.xml`, OG-изображения для мета-тегов).

Сейчас все изображения шаблона лежат в `musor/public/images/` — будут перемещены/заменены при очистке шаблона (см. `TODO.md`).

## Интернационализация (i18n)

Языки: **ru / uz / en** (русский — основной).

Подход: **собственная реализация без внешних библиотек** (без `i18next`, `react-intl` и т.п.).

- Словари — плоские JS-объекты в `src/locales/{ru,uz,en}.js`.
- Текущий язык и функция перевода — через React Context (`LanguageContext`) + хук `useLanguage()`.
- Выбор языка — компонент `LanguageSwitcher` (пишет в Context).

Обоснование: лендинг содержит десятки, а не тысячи строк; сложные возможности `i18next` (pluralization, ICU-формат, lazy loading namespaces) избыточны. Если в будущем потребуется плюрализация/сложная интерполяция — возможна миграция на `i18next` (замена Context на провайдер `i18next` + ключи в JSON).

## Управление состоянием

- **Локальное состояние** компонентов — `useState` / `useReducer`.
- **Глобальное состояние** — React Context API.
  - Сейчас только `LanguageContext`.
- **Внешние библиотеки (Redux, Zustand, Jotai, Recoil) не используются.**

Обоснование: объём глобального состояния мал (язык, возможно — открытое мобильное меню). Встроенных средств React достаточно.
